## P0 Bugs — Ship-Stopping (Verified)

### 1. `lru-cache` not installed when cache disabled, but always imported (cache)
**Status:** NEW — not in original report
**Files:** `lib/cache/lru.ts` always generated by `cache.module.ts`. `deps.ts` only installs `lru-cache` when `cfg.modules.cache`. 7+ modules import from `@/lib/cache`.
**Impact:** If cache module is disabled, `import LRU from "lru-cache"` throws `MODULE_NOT_FOUND` at runtime. **Every page crashes.**
**Fix:** Either always install `lru-cache`, or conditionally generate `lib/cache/` files, or use dynamic import with fallback.

### 2. `revalidateTag(tag, "page")` requires Next.js 16 (cache)
**Status:** NEW — not in original report
**Files:** `lib/cache/revalidate.ts` calls `revalidateTag(tag, "page")` with 2 arguments. Next.js 14/15 `revalidateTag` takes only 1 argument.
**Impact:** Runtime `TypeError: revalidateTag expects 1 argument, got 2` on every cache invalidation call.
**Fix:** Use single-argument `revalidateTag(tag)` for Next.js 14/15 compatibility, or add version detection.

### 3. SEO pages import from `@/lib/seo/` when SEO module disabled (ui-foundation + billing-core + blog + pwa)
**Status:** CONFIRMED — original P1 #3, upgraded to P0
**Files:** 4 ui-foundation pages (about/contact/terms/privacy), 3 billing-core pages (pricing/success/cancel), blog pages, PWA manifest route — all import `getPageMetadata` or `getSeoData` from `@/lib/seo/`.
**Impact:** When SEO module is disabled, `lib/seo/metadata.ts` and `lib/seo/jsonld.ts` are deleted by `backupAndRemove`. **TypeScript build crash.**
**Fix:** Conditionally generate these pages based on `ctx.modules.seo`, or use dynamic import with try/catch, or inline minimal fallbacks.

### 4. Welcome email call has parameter mismatch (auth-core)
**Status:** PARTIALLY FIXED — original P0 #6 evolved
**File:** `auth-core.module.ts` line 67-69 — calls `sendWelcomeEmail(user.email, user.name)` (2 positional args). But `sendWelcomeEmail` in `auth-emails.ts` expects `{ to: string; userName: string; dashboardUrl: string }` (object with 3 fields).
**Impact:** `TypeError: Cannot read properties of undefined (reading 'to')` or silent failure in try/catch. Welcome email never actually sends even when the function IS called.
**Fix:** Change call to `sendWelcomeEmail({ to: user.email, userName: user.name, dashboardUrl: env.NEXT_PUBLIC_APP_URL + "/app" })`.

### 5. Welcome email function is NEVER CALLED by any lifecycle hook (email-resend)
**Status:** CONFIRMED — original P0 #6
**File:** `sendWelcomeEmail` is defined in `auth-emails.ts` but no code path calls it. `patchAuthForEmail()` only wires `sendVerifyEmail` and `sendResetPasswordEmail`. Auth-core's `databaseHooks.user.create.after` calls it but with wrong params (see #4).
**Impact:** Welcome emails never send, regardless of parameter fix.
**Fix:** Wire `sendWelcomeEmail` into `patchAuthForEmail()` by adding a `databaseHooks.user.create.after` block or patching the existing one.

### 6. Dodo checkout orgId in returnUrl, NOT in webhook payload (billing-dodo)
**Status:** CONFIRMED — original P0 #3
**File:** `billing-dodo.module.ts` checkout route encodes `?orgId=X&userId=Y` in returnUrl. Dodo webhooks do NOT carry returnUrl query params back. `reconcileDodoPayload` extracts `org_id` from webhook `data` — it is always `null`.
**Impact:** Billing subscriptions stored with `org_id = NULL`. Billing page always shows "no subscription."
**Fix:** Either use Dodo's metadata API (if available) or extract orgId from returnUrl on the success page and store it there.

### 7. Webhook replay has NO idempotency guard (webhook-ledger)
**Status:** CONFIRMED — original P0 #16
**File:** `webhook-ledger.ts` — `webhookLedgerService_replay` calls `reconcileBillingEvent()` unconditionally, then writes `replayed_at` AFTER. No guard check for existing `replayed_at`.
**Impact:** Clicking "Replay" twice double-reconciles billing events (double-charges, duplicate subscriptions).
**Fix:** Add `if (row.replayedAt) return { alreadyReplayed: true }` BEFORE calling `reconcileBillingEvent`.

### 8. Jobs module: "inngest" and "cron-only" drivers generate IDENTICAL code (jobs)
**Status:** CONFIRMED — original P0 #26
**File:** `jobs.module.ts` — checks `!!ctx.modules.jobs?.enabled` but never checks `ctx.modules.jobs.driver`. Same 2 files generated regardless of driver value.
**Impact:** Users who select "inngest" get zero Inngest code. No scheduling, no cron, no automation.
**Fix:** Generate Inngest client + function + API route for "inngest" driver. Generate `vercel.json` crons for "cron-only".

### 9. Jobs module: No scheduling mechanism exists (jobs)
**Status:** CONFIRMED — original P0 #27
**Impact:** Reconcile endpoint exists but nothing calls it automatically. Must manually POST.
**Fix:** Generate `vercel.json` with `crons` array, or Inngest schedule, or external cron trigger.

### 10. Jobs module: Reconcile hardcodes "dodo" provider (jobs)
**Status:** CONFIRMED — original P1 #22
**File:** `jobs_runReconcile()` calls `webhookLedgerService_list({ provider: "dodo", limit: 50 })`. If billing is Stripe, always returns `{ attempted: 0, replayed: 0 }`.
**Fix:** Detect billing provider from config and filter accordingly, or iterate all providers.

### 11. OpenTelemetry is a phantom feature (observability)
**Status:** CONFIRMED — original P0 #25
**File:** `ctx.modules.observability.otel` exists in types but zero files check it or generate OTEL code.
**Impact:** Users who enable OTEL get nothing.
**Fix:** Implement or remove from schema.

### 12. No Sentry middleware for automatic error capture (observability)
**Status:** CONFIRMED — original P0 #24
**Impact:** Only `logger.error()` calls fire Sentry. Unhandled exceptions in API routes, server actions, and pages are NOT captured.
**Fix:** Add Next.js middleware or error.ts files with Sentry integration.

### 13. No `withSentryConfig` wrapping next.config.ts (observability)
**Status:** CONFIRMED — original P0 #14
**Impact:** Source maps not uploaded to Sentry. Production errors show minified stacks.
**Fix:** Generate `next.config.ts` wrapped with `withSentryConfig`.

### 14. `withServerCache` recreates `unstable_cache` on every call (cache)
**Status:** NEW — not in original report
**File:** `cache.module.ts` — `unstable_cache()` is called inside the returned async function, not at module scope.
**Impact:** L2 Next.js cache never warms. Every request executes the underlying function. L1 LRU cache is redundant. Caching is effectively broken.
**Fix:** Call `unstable_cache` once at module scope and export the wrapped function, or use `cache()` from React.

### 15. `patchAuthForEmail()` uses fragile regex string replacement (email-resend)
**Status:** CONFIRMED — original P0 #23
**Impact:** If user's `auth.ts` has different whitespace/formatting, replacements silently fail. No verification that patches succeeded.
**Fix:** Use AST-based modification (e.g., ts-morph) or composition API.

**Status:** CONFIRMED — original P2 #15, upgraded to P1

## Conclusion

The codebase has improved significantly since the original MODULE-GAP-ANALYSIS.md — **16 of 27 P0 bugs have been fixed**. However, **5 new critical issues** were discovered during this deep audit, primarily in the cache module and SEO import handling.

The most dangerous remaining issues are:
1. **Cache module conditional generation** — if a user disables caching, the entire app crashes
2. **SEO import build crashes** — disabling SEO breaks 10+ pages across 4 modules
3. **Webhook replay idempotency** — double-clicking replay causes double-billing
4. **Jobs module non-functional** — no scheduling, no driver differentiation, no automation
5. **Welcome email never fires** — defined but never wired into lifecycle

The codebase is in a **"beta-ready"** state — core auth, org switching, and billing status work, but edge cases, security hardening, and automation infrastructure need significant work before production deployment.
